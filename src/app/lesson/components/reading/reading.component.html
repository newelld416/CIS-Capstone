Background

The acronym ARIA stands for accessible rich internet applications, and it is set of html attributes that developers can use to give their users a more accessible experience. It supplements HTML so that interactions and widgets commonly used in applications can be passed to assistive technologies when there is not otherwise a mechanism. For example, ARIA enables accessible navigation landmarks in HTML4, JavaScript widgets, form hints and error messages, live content updates, and more.

To highlight an example of what aria elements can do for you application we have chosen to show the HTML markup for a progress bar. As you can see the element we have chosen is just a div element. By itself it is not able to tell a screen reader what the purpose of the element is and i most certainly can not tell any specific details about the element. So we have added our attribute on the element. The first element we have added is the role attribute. This atribute is used to define the overal purpose of the element. the next element is aria-valuenow and is used to show the current progress of the loading bar. The third and fourt key attributes are aria-valuemin and aria-valuemax and they are used to show the min and max values of the progress bar respectively. All of these elements are key examples of how we as developers can add key infomation to elements to help with accessibility needs. 

Incorrect
<div id="percent-loaded"></div>
Correct
<div id="percent-loaded" role="progressbar" aria-valuenow="75" aria-valuemin="0" aria-valuemax="100"></div>

For a more complete and live example please click on the "Live Example" button in the bottom left of your screen.

Standards

There are five key rules to keep in mind when it come to aria attributes.

  1. Use native elements where you can use them
  2. Do not change native semantics, unless you really have to
  3. All interactive ARIA controls must be usable with the keyboard
  4. Do not use role="presentation" or aria-hidden="true" on a visible focusable element
  5. All interactive elements must have an accessible name

Use native elements where you can use them

Essentially this rule is saying that if at all possible you should use native HTML elements when possible. Meaning that it is always possible to style a div element to whatever you want however if you want it to be a button you should be using the native HTML button element instead of making something else look like a button. This will save you lots of time as well as make your code more readable and maintainable.

Do not change native semantics, unless you really have to

This means when we have native elements like header elements we should leave those elements alone. Those elements often have default roles that are read by screen readers. If you must add a unique role to a native element like that you should add it in a wrapping tag like the example below. 

Incorrect
<h2 role=tab>heading tab</h2>
Correct
<div role=tab><h2>heading tab</h2></div>

All interactive ARIA controls must be usable with the keyboard

Efectively this rule states that if you give an HTML element a specifc role it should act like that role. For example, if using role=button the element must be able to receive focus and a user must be able to activate the action associated with the element using both the enter (on WIN OS) or return (MAC OS) and the space key.

Do not use role="presentation" or aria-hidden="true" on a visible focusable element

A general rule of thumb for showing and hiding elements is arai labels that describe that are not needed. When we dynamically show and hide elements the elements are added automatically. So explicitly defining them in the markup is not necesary.

All interactive elements must have an accessible name

There are certain elements we need to link through html attributes. The most common example of this is input forms. When we have an input element we often acompany that input box with a label. Now there are several ways that we can do that but as long as they are created as separate elements we need to link them. Below you can see two examples that are both syntatically correct however as you can see in the second example the two HTML elements are linked. 

Incorrect
<label>user name</label> <input type="text">
Correct
<label for="uname">user name</label> <input type="text" id="uname">

Background

Contrast is defined as the state of being strikingly different from something else in juxtaposition or close association. Which means that in order to have good contrast we need to have different colors placed close to each other when designing and developing our applications. This is important for a wide variety of impairments such as color blindness, visual deficiencies and even screens with poor color resolution.

The overall goal as with most accessibility examples we will talk about in this lesson is to provide a clear experience visually for your application.

There are many types of users that benefit from an experience that follows this contrast requirement. Among them being individuals with low vision, individuals with low contrast vision or individuals with color vision deficiency.

Standards

The standard contrast ratio is 4.5:1, meaning if the background is a solid color (or all black or all white) then the relative luminance of the text can be maintained by making sure that each of the text letters have 4.5:1 contrast ratio with the background. Now let’s break down what that means. If I have an all-black (#000000) background that means the text on that background needs to be at least 4.5 times brighter.

Also included in this standard is Images. If the text of an image is meant to convey a message or meant to be understood as text, then it must meet the same accessibility standard of 4.5: 1. A good example of this is a company showing its logo as a jpg file in the header. The logo is meant to be subscribed as text and should therefore follow the contrast accessibility guideline of 4.5:1. An exception to this rule would be if a website had an image of a neighborhood landscape and in that landscape was the address of a house in the background. This is text on your webpage, however since it is not meant to convey a meaningful message to the user you do not have to worry about meeting accessibility guidelines.

As developers and designers there are things, we can do to help mitigate contrast issues. Larger text has wider character strokes and can be much easier for the user to read at a lower contrast ratio. So, in examples where colors cannot be changed for a specific reason the design can implement a view with larger text to help alleviate the concern. With larger text (18 point or larger, or 14 point or larger and bold) you could use a contrast ration of 3:1 that would be acceptable for most accessibility guidelines. As always with these guidelines there are not many had and fast rules and we as implementors can be creative about how solve problems for our users. We just have to be cognizant of all of our users when we consider how we may solve a problem.

Background

Color usage is extremely important when it comes to accessibility. It can be used to differentiate boundaries as well as convey messages. But as with every accessible feature the goal is to ensure that all users can access whatever information is being conveyed by the color choices.

When we think about color we often think about supporting people who experience “Color Blindness” but there are many other type of individuals that need to be considered, like the following:

  • Users with partial sight often experience limited color vision
  • Some older users may not be able to see color well
  • Users who have color-blindness benefit when information conveyed by color is available in other visual ways
  • People using text-only, limited color, or monochrome displays may be unable to access color-dependent information
  • Users who have problems distinguishing between colors can look or listen for text cues
  • People using Braille displays or other tactile interfaces can detect text cues by touch

Background

The most important feature when it comes to interactive elements is to make them easily identifiable. You need to make sure that whatever disability your user is working with they understand that a button is a button, and a link is a link, and a form element is a form element. Now there are an infinite number of ways to accomplish this and there are new designs coming out daily that make it easier and easier for a user to identify these elements. In this lesson we will talk about some of the common failures we see across many sites and some of the easiest ways to drive a consistent and easily recognizable experience.

Here is a list of some common interactive elements that we see often can cause accessibility issues

  • Using <div> elements that act like <button> elements
  • Using input fields without an attached label
  • Using only one method to provide user feedbak
  • Using mouse interactions solely to convey focus

Standards

The first failure that we talked about was using generic HTML elements to accomplish a task that can be solved more easily with native HTML elements. The classic example that comes up is using elements like <a> and <span> to accomplish what is needed by a button element. This inherently causes isses for tools like screen readers that are relient on these native elements to understand the purpose of the element. There are always things you can do to make them act hte same way however you are always going to ad more work that is unnecesssary to the orginal goal of the element. Below you can see a list of things you would need to consider if using an <a> element to act as a button.

  • <button> has a button role, <a href> has a link role
  • <a href=”#” role=”button”> only works with click and enter key, <button> works with spacebar also
  • Focus jumps to top of page when user activates an <a href=”#”> control whereas focus stays put on a <button>
  • <button> looks like a Button, <a href> looks like a Link
  • If you forget the href attribute, i.e. <a onclick> then there is NO role, NO keyboard focusability, and NO enter key activation. <a> is basically the same as a <span> unless you have the href attribute

The second failure that we spoke about was one that we have already covered in our lesson about ARIA elements but we will show the example again as a refresher. 

Incorrect
<label>user name</label> <input type="text">
Correct
<label for="uname">user name</label> <input type="text" id="uname">

Another very simple mistake we make when it comes to interactive elements is making sure we consicer users who are unable or unwilling to use a mouse. The key area we miss is the difference we miss between hover and focus. Hover is inacted when a users mouse hovers overa specific element. It allows us to change the visualization of that element often times showing that it is clickable or that the user can take action against it. What we miss as developers is when a user is not using a mouse, we also need to take into account that the same visualization que is taken when the element obtains focus. This is done through the user using non mouse capabilities to navigate your expereience. For example when a user tabs through a form how do we show the users which element is in focus in the same manner that we would if the hovered of those elements. As an example below you cann see how both CSS selectors are used to describe the styling. 

a:focus, a:hover {
  color: white;
  background-color: black;
  text-decoration: none
}

Standards

To support best practices around navigation there are some key elements we need to consider, structure and styling. Structure referes to the semantic markup that conveys the menu structure to users. Menus coded semantically can easily adapt to different situations, such as small screen displays, screen magnification, and other assistive technology. While Style is the act of making something look the way a user expects it to look.

Structure

It is important to use key native elements when developing the structure of you navigation cability. This wil ensure that your navigation elements work well under multiple conditions as well as work well with assistive technologies such as screen readers. When it comes to styling your navigation components there are elements in which you can be createive but it is important to keep in mind that users expect to see thing in a logical manner.
